# Requesting a Feature for LeoSetter

We love hearing ideas from the community! This page explains how to submit a feature request, what makes a great one, and what happens after you submit it.

---

## Step 1 — Check for existing requests

Before opening a new request, search the [existing issues](https://github.com/AHJ32/LeoSetter/issues?q=is%3Aissue+label%3Aenhancement) to see if someone has already suggested the same thing. If they have, add a 👍 reaction to the existing issue and leave a comment with any additional context you can offer — this helps us prioritise.

---

## Step 2 — Open a Feature Request

Click the link below to open a pre-filled feature request template:

> **[🚀 Open a Feature Request →](https://github.com/AHJ32/LeoSetter/issues/new?template=feature_request.md)**

---

## Step 3 — Describe your idea clearly

A well-described request is much more likely to be accepted and implemented quickly. When writing your request, include:

| Field | What to write |
|---|---|
| **Title** | A short, action-oriented summary (e.g. "Add bulk rename based on EXIF date") |
| **What the feature does** | A clear description of the behaviour you want |
| **Why it is useful** | Explain the real-world use case — who benefits and how |
| **How you imagine it working** | Describe the UI or workflow, even roughly (optional but very helpful) |
| **Screenshots / mockups** | Attach any visuals that clarify your idea (optional) |

### ✅ Example of a good request

> **Title:** Export metadata to a CSV file
>
> **What:** Add a "Export to CSV" button in the toolbar that writes the current metadata of all loaded images to a `.csv` file.
>
> **Why:** I manage large photo archives and need to audit metadata in spreadsheet form before editing. Currently I have to do this manually with ExifTool on the command line.
>
> **How:** A save dialog could let me choose the output path. Each row in the CSV would be one image, and each column would be a metadata field.

### ❌ Example of a request that is too vague

> **Title:** Make it better
>
> **What:** The app should have more features and be easier to use.

Vague requests without a concrete use case are difficult to evaluate and unlikely to move forward.

---

## Step 4 — What happens next?

After you submit your request:

1. **A maintainer will review it** — usually within a few days
2. **It will be labelled** — common labels are:
   - `enhancement` — confirmed valid feature idea
   - `accepted` — approved and open for implementation (by you or anyone else!)
   - `needs-info` — we need more detail before deciding
   - `wontfix` — out of scope or conflicts with the project direction
3. **If marked `accepted`**, the issue is open for anyone to implement. See [CONTRIBUTING.md](../CONTRIBUTING.md) if you'd like to build it yourself.
4. **Discussion** — feel free to continue discussing the idea in the issue comments. Collaboration often leads to better solutions.
